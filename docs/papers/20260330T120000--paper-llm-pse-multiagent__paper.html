问题

LLM 已经能写代码、做数学推理、处理自然语言。但在过程系统工程（PSE）这种硬核工程领域——涉及物理约束、微分方程、参数估计、实时控制——通用 LLM 到底能不能用？多 Agent 协作比单 Agent 好多少？

这篇论文做了一件很直接的事：拿结晶过程（crystallization）作为试金石，让四个 LLM 系统（两个多 Agent、两个单 Agent）在三个真实 PSE 任务上对比：

  1. 软传感器标定：从红外光谱预测药物浓度
  2. 机理建模：建立结晶过程的 population balance model（PBM），拟合实验数据
  3. 非线性模型预测控制（NMPC）：设计控制器调温度，控制晶体大小和质量

选结晶是因为它特别难——强非线性、紧约束、群体平衡动力学耦合，是 LLM 容易翻车的场景。

翻译

一个类比：四个实习生做化工设计

想象你是一个化工厂的总工程师，需要完成一个结晶工艺的全套设计（传感器标定 → 动态模型 → 控制器）。你有四个「实习生」可以选：

Dyad 2.1（多 Agent + 专用建模语言）

类比

这位实习生自带了一套专业工具箱（Julia 建模语言、编译器、调试器）。他的工作方式是：先查手册确认语法，写代码，编译测试，发现问题就用「因果追踪」一步步定位 bug 的根源——不是猜，而是沿着方程图逆向排查。

Claude Opus 4.6（多 Agent + 通用工具）

类比

这位实习生很聪明，没有专用工具但能用 Python/Julia 写代码。他会尝试多种模型结构（Arrhenius-only、secondary-nucleation-only、组合），选最好的。但有时候会过拟合——在训练集上很好看，泛化到新工况就飘了。

ChatGPT 5.2 / Gemini Pro 3（单 Agent）

类比

这两位实习生一个人干所有事——建模、拟合、调参都在一轮对话里完成。他们经常在最开始就把模型结构定死了，不会回头检查物理合理性。结果有时候参数数值离谱（比如生长速率 kg 差了 5 个数量级），或者模型预测在近饱和区域发散。

核心发现：多 Agent 系统（Dyad 和 Claude Opus）把任务分解成子任务（建模 → 验证 → 参数估计），每步都有物理约束检查。单 Agent 系统在一个 prompt 里做所有事，容易「丢掉」物理直觉。

架构

三个案例的技术细节

Case 1: ATR-FTIR 光谱标定

任务：从 247 条红外光谱 + 温度 + 溶剂组成 → 预测扑热息痛摩尔分数

所有 LLM 都不约而同地选择了潜变量线性回归（PCR 或 PLSR）——这是光谱标定的标准方法。有趣的是差异出在预处理上：

系统方法预处理测试集 R²
Dyad 2.1PCR (10 components)标准化~0.99
Claude Opus 4.6PLSR (6 components)Savitzky-Golay + SNV~0.98
ChatGPT 5.2PLSR (7 components)Savitzky-Golay + 标准化~0.99
Gemini Pro 3PLSR (15 components)10-fold CV 选组分数~0.99 (最佳)

结论：对于这种"教科书级"的标定任务，四个 LLM 都能搞定。差异主要在预处理策略上。

Case 2: 结晶动力学建模（PBM）

任务：建立硫酸钾结晶的群体平衡模型（Population Balance Model），拟合溶解和结晶实验数据

所有 LLM 都恢复了相同的 PBM 结构——矩量法（Method of Moments）：

dμ₀/dt = B (成核速率) dμ₁/dt = G·μ₀ (一阶矩 = 总长度) dμ₂/dt = 2G·μ₁ (二阶矩 = 总面积) dμ₃/dt = 3G·μ₂ (三阶矩 = 总体积) dC/dt = -3ρ·kᵥ·G·μ₂ (溶质质量守恒) B = 成核速率 (nucleation rate) G = 生长速率 (growth rate) C = 溶质浓度, Cₛ = 饱和浓度 S = C/Cₛ (过饱和度)

但成核和生长速率的表达式差异巨大——这是区分好坏的关键：

系统成核假设验证 R²物理一致性
Dyad 2.1Arrhenius 修正 + 二次成核 + 衰减项高最佳（长时间达平衡）
Claude Opus 4.6Arrhenius 修正 + μ₃ 依赖成核高近饱和区域有残余漂移
ChatGPT 5.2仿射尺寸依赖 + 两项模型低（μ₀ 严重偏差）差
Gemini Pro 3Arrhenius 修正 + 晶体群依赖低（μ₁ 负 R²）差
多 Agent 系统（Dyad, Claude Opus）将建模分解为子任务（模型结构 → 参数估计 → 验证 → 迭代），每步有物理约束检查。单 Agent 系统在一个 pass 里做所有决策，导致参数数值偏差巨大（ChatGPT 的 kg = 2993 vs. Dyad 的 kg = 86.9）。

Case 3: NMPC 控制器设计

任务：设计 NMPC 控制器，通过调节温度来控制晶体平均尺寸 L̄₁₀ = μ₁/μ₀ 和质量 m

三个 LLM（Dyad, Claude Opus, Gemini Pro）各自提出了 NMPC 调参方案，ChatGPT 的模型预测不准所以被排除：

参数Dyad 2.1Claude Opus 4.6Gemini Pro 3
预测时域 P202010
终端惩罚 θ₁, θ₂200, 5005, 50, 0
移动抑制 λ₁500.0110
温度变化限制 ΔT±0.3~0.5 K±1 K±1 K

结果：Dyad 在 5 个工况中总体表现最好——L̄₁₀ 和 m 的偏差最小且最一致，控制量也最保守。所有 NMPC 的求解时间都在 1 分钟采样周期内，说明实时可行。

评价

⚖️ 这是目前最全面的「LLM 做化工」基准测试——从标定到建模到控制的完整 PSE 闭环。多 Agent 系统（特别是带专用建模语言的 Dyad）在需要物理一致性和迭代验证的任务上显著优于单 Agent。

亮点

  • 真实工业场景：不是玩具问题，是真实的结晶实验数据和设备
  • 完整 PSE 闭环：标定 → 建模 → 验证 → 模型更新 → 控制器设计，五步全覆盖
  • 公平对比：4 个 LLM 系统、相同初始 prompt、代码全部开源
  • NMPC 实时可行：1 分钟采样周期内求解，工业可部署
  • Dyad 作为开源工具：带有专用建模语言 + 因果调试 + 编译验证的完整系统

局限

  • 只测试了结晶过程——其他化工单元（精馏、反应器）的泛化性未验证
  • Dyad 依赖 Julia 生态（ModelingToolkit.jl），Python 用户的迁移成本高
  • 多 Agent 系统的交互时间长（Dyad 需要 ~2 小时收敛，单 Agent ~20 分钟）
  • NMPC 只测试了 5 个工况，未涉及扰动鲁棒性和约束违反分析

一句话总结

多 Agent LLM 系统（特别是带专用建模语言和物理约束验证的 Dyad）在过程系统工程的三个核心任务（标定、建模、NMPC 控制）上系统性地优于单 Agent 通用 LLM，关键优势在于任务分解和物理一致性检查。
